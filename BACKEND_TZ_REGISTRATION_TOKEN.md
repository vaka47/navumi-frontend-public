# Техническое задание: Отложенная авторизация до создания профиля

## Цель
Изменить процесс регистрации так, чтобы пользователь не получал активную сессию до завершения создания профиля. Вместо сессии использовать временный токен для прохождения этапов регистрации.

---

## 1. Изменения в API `/api/confirm-code/`

### Текущее поведение
- После подтверждения кода создается активная сессия (sessionid)
- Пользователь сразу получает доступ к защищенным ресурсам

### Новое поведение
- После подтверждения кода **НЕ создавать** активную сессию
- Создать временный токен регистрации (registration token)
- Вернуть токен в ответе

### Формат запроса
```http
POST /api/confirm-code/
Content-Type: application/json
X-CSRFToken: <csrf_token>

{
  "email": "user@example.com",
  "code": "12345"
}
```

### Формат ответа (успех)
```json
{
  "success": true,
  "registration_token": "reg_abc123def456ghi789jkl012mno345pqr678stu901vwx234yz",
  "expires_at": "2025-12-02T15:30:00Z"
}
```

### Формат ответа (ошибка)
```json
{
  "error": "Неверный код подтверждения"
}
```

### Требования к токену
- **Формат**: Строка длиной 40-64 символа, безопасная для URL (base64url или hex)
- **Префикс**: `reg_` для идентификации типа токена
- **Время жизни**: 15 минут с момента создания
- **Хранение**: В базе данных или Redis с полями:
  - `token` (уникальный)
  - `email` (связанный email)
  - `created_at` (время создания)
  - `expires_at` (время истечения)
  - `used` (boolean, помечается как использованный после создания профиля)
- **Безопасность**: 
  - Токен должен быть криптографически случайным
  - Одноразовый (после использования помечается как `used`)
  - Автоматически удаляется после истечения срока

## 2. Изменения в API `/api/choose-role/`

### Текущее поведение
- Принимает запрос с активной сессией
- Сохраняет выбор роли в сессии/БД

### Новое поведение
- Принимает `registration_token` вместо проверки сессии
- Валидирует токен (существует, не истек, не использован)
- Сохраняет выбор роли, связанный с токеном
- Возвращает обновленный токен (опционально, если нужно продлить срок)

### Формат запроса
```http
POST /api/choose-role/
Content-Type: application/json
X-CSRFToken: <csrf_token>

{
  "registration_token": "reg_abc123def456ghi789jkl012mno345pqr678stu901vwx234yz",
  "role": "club"  // или "client"
}
```

### Формат ответа (успех)
```json
{
  "success": true,
  "redirect": "/auth/create-club-profile?token=reg_abc123def456ghi789jkl012mno345pqr678stu901vwx234yz"
}
```

### Формат ответа (ошибка)
```json
{
  "error": "Токен регистрации недействителен или истек"
}
```

### Валидация
- Проверить наличие токена в БД
- Проверить, что токен не истек (`expires_at > now()`)
- Проверить, что токен не использован (`used == false`)
- Проверить, что `role` равен `"club"` или `"client"`

## 3. Изменения в API `/api/create-club-profile/`

### Текущее поведение
- Принимает запрос с активной сессией
- Создает профиль клуба
- Возвращает редирект

### Новое поведение
- Принимает `registration_token` в теле запроса или query параметре
- Валидирует токен
- Создает профиль клуба
- **Создает активную сессию** для созданного пользователя
- Помечает токен как использованный (`used = true`)
- Возвращает редирект на страницу профиля

### Формат запроса
```http
POST /api/create-club-profile/?token=reg_abc123def456ghi789jkl012mno345pqr678stu901vwx234yz
Content-Type: multipart/form-data
X-CSRFToken: <csrf_token>

FormData:
  - username: "clubname"
  - name: "Club Name"
  - phone: "+1234567890" (опционально)
  - profile_picture: <File> (опционально)
```

**Альтернативный вариант** (если токен в теле):
```http
POST /api/create-club-profile/
Content-Type: multipart/form-data
X-CSRFToken: <csrf_token>

FormData:
  - registration_token: "reg_abc123def456ghi789jkl012mno345pqr678stu901vwx234yz"
  - username: "clubname"
  - name: "Club Name"
  - phone: "+1234567890" (опционально)
  - profile_picture: <File> (опционально)
```

### Формат ответа (успех)
```json
{
  "success": true,
  "redirect": "/clubname",
  "username": "clubname"
}
```

### Формат ответа (ошибка)
```json
{
  "error": "Токен регистрации недействителен или истек"
}
```
или
```json
{
  "error": "Имя пользователя уже занято"
}
```

### Валидация
- Проверить наличие токена в БД
- Проверить, что токен не истек
- Проверить, что токен не использован
- Проверить, что для токена уже выбран `role: "club"`
- Валидировать данные профиля (username, name, и т.д.)
- После успешного создания:
  - Создать активную сессию для пользователя
  - Пометить токен как `used = true`
  - Удалить токен из БД (или оставить для аудита)

## 5. Новый API для проверки токена (опционально)

### Назначение
Позволяет фронтенду проверить валидность токена перед отправкой формы

### Формат запроса
```http
GET /api/validate-registration-token/?token=reg_abc123def456ghi789jkl012mno345pqr678stu901vwx234yz
```

### Формат ответа (валидный токен)
```json
{
  "valid": true,
  "email": "user@example.com",
  "role": "club",  // или "client", или null если роль не выбрана
  "expires_at": "2025-12-02T15:30:00Z"
}
```

### Формат ответа (невалидный токен)
```json
{
  "valid": false,
  "error": "Токен недействителен или истек"
}
```

## 6. Обработка ошибок

### Общие ошибки для всех endpoints
- **400 Bad Request**: Неверный формат запроса, отсутствуют обязательные поля
- **401 Unauthorized**: Токен недействителен, истек или уже использован
- **403 Forbidden**: Попытка создать профиль с ролью, отличной от выбранной
- **409 Conflict**: Username уже занят
- **500 Internal Server Error**: Ошибка сервера

### Специфичные ошибки
- `/api/confirm-code/`:
  - `"Неверный код подтверждения"`
  - `"Код истек"`
  - `"Email не найден"`
  
- `/api/choose-role/`:
  - `"Токен регистрации недействителен или истек"`
  - `"Неверная роль. Допустимые значения: 'club', 'client'"`
  
- `/api/create-club-profile/` и `/api/create-client-profile/`:
  - `"Токен регистрации недействителен или истек"`
  - `"Роль не выбрана. Сначала выберите роль"`
  - `"Попытка создать профиль с неверной ролью"`
  - `"Имя пользователя уже занято"`
  - `"Имя пользователя содержит недопустимые символы"`
  - `"Имя пользователя зарезервировано"`

## 7. Безопасность

### Требования
1. **Токены должны быть криптографически случайными**
   - Использовать `secrets.token_urlsafe()` или аналоги
   - Минимум 32 байта энтропии

2. **Ограничение времени жизни**
   - Токен действителен 15 минут
   - Автоматическая очистка истекших токенов (cron job или TTL в Redis)

3. **Одноразовость**
   - После успешного создания профиля токен помечается как `used`
   - Повторное использование токена запрещено

4. **Rate limiting**
   - Ограничить количество попыток подтверждения кода (например, 5 попыток в минуту)
   - Ограничить количество запросов с токеном (например, 10 запросов в минуту)

5. **Валидация CSRF**
   - Все POST запросы должны проверять CSRF токен

6. **Логирование**
   - Логировать все попытки использования токенов
   - Логировать ошибки валидации
   - Логировать успешное создание профилей

## 9. Обратная совместимость

### Важно
- Если фронтенд еще не обновлен, старые запросы с сессией должны обрабатываться
- Можно добавить проверку: если есть валидная сессия и нет токена, использовать старую логику
- После обновления фронтенда старую логику можно удалить

### План миграции
1. Добавить поддержку токенов (новый код)
2. Оставить поддержку сессий (старый код) на период миграции
3. Обновить фронтенд
4. Удалить поддержку сессий в регистрации

## 10. Тестирование

### Тест-кейсы
1. ✅ Подтверждение кода → получение токена
2. ✅ Выбор роли с валидным токеном
3. ✅ Выбор роли с истекшим токеном → ошибка
4. ✅ Создание профиля клуба с валидным токеном → создание сессии
5. ✅ Создание профиля клиента с валидным токеном → создание сессии
6. ✅ Попытка создать профиль с неверной ролью → ошибка
7. ✅ Попытка использовать токен повторно → ошибка
8. ✅ Истечение токена → автоматическая очистка
9. ✅ Валидация токена через `/api/validate-registration-token/`

## 11. Примеры использования

### Полный flow регистрации

1. **Подтверждение email:**
```http
POST /api/confirm-code/
{
  "email": "user@example.com",
  "code": "12345"
}

Response:
{
  "success": true,
  "registration_token": "reg_abc123...",
  "expires_at": "2025-12-02T15:30:00Z"
}
```

2. **Выбор роли:**
```http
POST /api/choose-role/
{
  "registration_token": "reg_abc123...",
  "role": "club"
}

Response:
{
  "success": true,
  "redirect": "/auth/create-club-profile?token=reg_abc123..."
}
```

3. **Создание профиля:**
```http
POST /api/create-club-profile/?token=reg_abc123...
FormData:
  - username: "myclub"
  - name: "My Club"
  - profile_picture: <File>

Response:
{
  "success": true,
  "redirect": "/myclub",
  "username": "myclub"
}
```

После этого пользователь получает активную сессию и может использовать приложение.
